  1. Résumé de la nouvelle logique corrigée

  Une plate-forme unique de gestion des demandes (ancien "réclamations") traitée
   par une équipe de confirmatrices formant une agence globale. Trois acteurs :

  - Client : signale un problème ou répond à une demande livreur. Il est le seul
   à pouvoir déclencher une reprogrammation.
  - Livreur : signale un incident terrain avec des motifs propres à son métier.
  Trois tentatives échouées sur une commande déclenchent une remontée
  automatique, quel que soit le motif.
  - Confirmatrice : traite une file globale unique, priorisée par un score
  intelligent (ancienneté, tentatives, urgence). Plus de frontière par
  gouvernorat.

  Les demandes suivent un cycle simple à quatre statuts professionnels (Envoyée
  → En cours de traitement → Clôturée / Refusée). Les cas de modification
  d'adresse sont traités via une capture GPS + carte interactive, applicable par
   un seul bouton.

  3. Nouvelle logique recommandée pour les statuts

  Quatre statuts, cycle strictement linéaire :

  1. Envoyée — la demande vient d'être créée, elle est dans la file globale,
  aucune confirmatrice n'a encore cliqué dessus.
  2. En cours de traitement — une confirmatrice a cliqué "Prendre en charge". Un
   verrou logique évite qu'une autre la traite en parallèle.
  3. Clôturée — la demande est résolue (correction appliquée, échange créé,
  livraison faite, etc.).
  4. Refusée — la demande n'est pas recevable (doublon, motif invalide, preuve
  insuffisante). Motif de refus obligatoire.

  Règles :
  - Une demande ne peut revenir en arrière (pas de "réouverture"). Si un nouveau
   cas apparaît, on crée une nouvelle demande liée à la commande.
  - Clôturée = visible en historique, plus d'actions.
  - Deux statuts "ouverts" (Envoyée + En cours de traitement), deux statuts
  "fermés" (Clôturée + Refusée).

  4. Nouvelle logique recommandée pour le changement d'adresse

  Ce flux s'applique uniquement dans deux cas :
  - Le client crée une réclamation avec motif CHANGEMENT_ADRESSE.
  - Le client répond à une demande livreur de type ADRESSE_INCORRECTE,
  ADRESSE_INCOMPLETE ou ADRESSE_INTROUVABLE.

  Écran unifié, trois zones :

  1. Bloc information : rappelle la commande concernée, l'ancienne adresse, qui
  demande la correction (soi-même ou le livreur).
  2. Bloc saisie adresse :
    - Champ texte adresse (obligatoire, pour la lisibilité humaine)
    - Bouton principal "Prendre mon adresse actuelle" → capture
  latitude/longitude via GPS, pin posé automatiquement sur la carte, zoom
    - Widget carte interactive en dessous : le client peut déplacer le pin s'il
  veut corriger l'emplacement
    - Affichage textuel des coordonnées capturées (pour confiance)
  3. Bouton final : "Appliquer" → envoie adresse + latitude + longitude au
  backend. Le backend stocke les 3 sur la demande, et seule la confirmatrice
  peut propager vers le profil/commande en cliquant sur "Appliquer la
  correction".

  Côté confirmatrice, elle voit :
  - L'ancienne adresse
  - La nouvelle adresse texte
  - Un mini-aperçu de carte centré sur les coordonnées (lecture seule)
  - Un seul bouton "Appliquer la correction"

  Ce bloc carte est masqué pour tous les autres motifs (changement de numéro,
  annulation, colis endommagé, etc.).

  5. Nouvelle logique recommandée pour la répartition / attribution des 
  commandes entre confirmatrices

  Principe : une seule agence virtuelle, pas de frontière gouvernorat. Toutes
  les confirmatrices voient une file commune.

  5.1 Score de priorité (tri de la file)

  Chaque demande reçoit un score calculé dynamiquement :

  ┌────────────────────────────────────────────────────┬────────────────────┐
  │                      Facteur                       │  Poids indicatif   │
  ├────────────────────────────────────────────────────┼────────────────────┤
  │ Ancienneté de la demande (minutes depuis création) │ +1 point / 10      │
  │                                                    │ minutes            │
  ├────────────────────────────────────────────────────┼────────────────────┤
  │ Nombre de tentatives livreur ≥ 3                   │ +50 points         │
  │                                                    │ (booster fort)     │
  ├────────────────────────────────────────────────────┼────────────────────┤
  │ Motif urgent (COLIS_ENDOMMAGE, CLIENT_REFUSE,      │ +30 points         │
  │ COLIS_NON_CORRESPONDANT)                           │                    │
  ├────────────────────────────────────────────────────┼────────────────────┤
  │ Créneau de livraison dépassé                       │ +20 points         │
  ├────────────────────────────────────────────────────┼────────────────────┤
  │ Source = LIVREUR (bloque une livraison en cours)   │ +10 points         │
  └────────────────────────────────────────────────────┴────────────────────┘

  La file est triée par score décroissant. Un badge "URGENT" s'affiche au-delà
  d'un certain seuil.

  5.2 Score de disponibilité (pour router la notification de création)

  Quand une demande est créée, on notifie en priorité la confirmatrice avec le
  meilleur score de disponibilité :

  ┌──────────────────────────────────────────┬──────────────────────────────┐
  │                 Facteur                  │            Effet             │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ Statut connexion (en ligne / hors ligne) │ exclut les hors ligne        │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ Nombre de cas "En cours de traitement"   │ pénalise                     │
  │ sur elle                                 │                              │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ Cas clôturés dans les 30 dernières       │ bonus (elle est productive)  │
  │ minutes                                  │                              │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ Inactivité > 15 min                      │ pénalise (probablement en    │
  │                                          │ pause)                       │
  └──────────────────────────────────────────┴──────────────────────────────┘

  Cette notification est indicative, non exclusive : la demande reste visible
  pour toutes les confirmatrices dans la file globale.

  5.3 Flux concret

  1. Confirmatrice se connecte → voit la file globale triée par score.
  2. Bouton "Suivant" en haut → pioche automatiquement le cas le mieux
  prioritaire non encore pris en charge.
  3. Ou sélection manuelle dans la liste.
  4. "Prendre en charge" → verrouille la demande sur elle et passe en "En cours
  de traitement".
  5. Si une demande "En cours de traitement" n'a pas eu d'action pendant 30 
  minutes, elle redevient disponible dans la file (timer silencieux).
  6. Admin voit une vue supplémentaire : charge par confirmatrice, temps moyen
  de traitement, cas re-libérés. Utile pour monitoring.

  Cette logique est simple à expliquer, moderne dans le ressenti, et 
  auto-régulée. Pas de machine learning, pas de règles obscures.

  8. Logique des tentatives et remontée vers la confirmatrice

  8.1 Définition d'une tentative

  Une tentative = une journée d'échec de livraison sur une commande, quel que
  soit le motif. Règle anti-spam : un seul enregistrement par couple (commande, 
  jour calendaire) — si le livreur change de statut plusieurs fois le même jour,
   on garde le dernier motif et on ne recompte pas.

  8.2 Création de la demande

  - Motif différé (CLIENT_NON_JOIGNABLE, CLIENT_ABSENT) : aucune demande n'est
  créée aux tentatives 1 et 2. À la tentative n° 3, une demande est créée
  automatiquement avec badge "URGENT — 3 tentatives échouées".
  - Motif immédiat (adresse, numéro, refus, colis endommagé, autre) : demande
  créée dès la tentative n° 1. Le compteur continue de monter.

  8.3 Règle transversale : seuil 3 tentatives

  À partir de 3 tentatives sur la même commande (peu importe le motif), qu'une
  demande existe déjà ou pas :
  - La demande concernée reçoit le booster de priorité +50 points → remonte en
  tête de la file globale.
  - Elle reçoit un badge rouge "3 TENTATIVES — À TRAITER".
  - Une notification push est envoyée à la confirmatrice la plus disponible.
  - Dans l'onglet "À traiter" de la confirmatrice, un filtre rapide "3+ 
  tentatives" est visible en haut pour isoler ces cas.

  8.4 Côté livreur

  Le livreur voit sur sa fiche de livraison : "Tentative 2/3 — au 3e échec le
  cas remonte au support". Transparent et dissuasif sur le sérieux du suivi.

  1. Résumé corrigé de la logique finale

  Une plateforme de gestion à deux flux visibles :

  - Réclamations — initiées par le client, dirigées vers le support (livreur ou
  confirmatrice).
  - Demandes — initiées par le livreur, dirigées vers le client lorsqu'une
  action est attendue de lui.

  Et un troisième flux interne, jamais visible par le client mais pleinement
  géré par la confirmatrice : les Incidents livreur — remontées livreur →
  confirmatrice sans action client (refus, autre incident, tentatives répétées).

  Les quatre statuts métier sont uniformes : Envoyée, En cours de traitement, 
  Clôturée, Refusée. Ils s'appliquent à chaque cas quel que soit le flux.

  Côté client, un code couleur rouge / vert est ajouté uniquement sur les
  demandes reçues du livreur pour rendre immédiatement visible si la correction
  est à fournir ou déjà envoyée. Ce rouge/vert n'est pas un statut métier, c'est
   un indicateur visuel UX.

  La confirmatrice traite une file globale unique, triée par ordre de priorité
  simple : urgents → tentatives répétées → anciens. Elle peut traiter les cas de
   tout gouvernorat, et peut changer à la fois le statut du cas et le statut de
  la commande rattachée.

  La reprogrammation est exclusivement client : aucun acteur ne peut choisir une
   nouvelle date à la place du client.

  5. Cas qui créent une demande au client

  Uniquement les motifs A (Adresse incorrecte, Numéro incorrect).

  Cycle de vie d'une demande livreur → client

  1. Livreur coche Adresse incorrecte ou Numéro incorrect sur la commande.
  2. Une Demande est créée immédiatement.
  3. Statut métier initial : Envoyée.
  4. Indicateur visuel client : rouge — « Correction requise ».
  5. Client ouvre sa demande dans son espace, clique sur le bouton de
  correction, renseigne la nouvelle adresse (avec carte GPS) ou le nouveau
  numéro, clique sur Appliquer.
  6. Au moment du clic « Appliquer » :
    - Le statut métier passe automatiquement à En cours de traitement.
    - L'indicateur visuel client passe au vert — « Correction envoyée ».
    - La confirmatrice reçoit une notification : un cas attend validation.
  7. Confirmatrice ouvre le détail, vérifie la correction proposée, clique sur
  Appliquer la correction pour propager la modification au profil client et/ou à
   la commande. Puis clique sur Clôturer — le statut métier passe à Clôturée.
  8. Si la correction est refusée (suspect, incohérent) → Refuser avec motif →
  statut Refusée.

  6. Cas qui vont directement à la confirmatrice

  Motifs B — Refus client et Autre incident.

  Cycle de vie

  1. Livreur coche le motif.
  2. Un Incident est créé immédiatement.
  3. Statut métier initial : Envoyée.
  4. Aucune visibilité côté client, aucun indicateur rouge/vert.
  5. Le cas apparaît dans la file confirmatrice avec badge « Incident livreur ».
  6. La confirmatrice clique sur Prendre en charge → statut En cours de 
  traitement.
  7. Elle traite (appel client, appel livreur, consultation commande), puis :
    - Clôturer → Clôturée
    - Refuser → Refusée (avec motif de refus)

  ---
  7. Cas qui vont à la confirmatrice après 3 tentatives

  Motifs C — Téléphone fermé, Client non joignable, Client absent.

  Logique de comptage

  - Une tentative = une journée d'échec sur une commande (unicité par couple
  commande + jour calendaire pour éviter le spam).
  - Aux tentatives 1 et 2 : aucun cas créé. Le livreur voit « Tentative 1/3 »
  puis « Tentative 2/3 » sur sa fiche.
  - À la tentative 3 : un Incident est créé automatiquement, statut métier
  Envoyée, avec le badge rouge « 3 tentatives échouées ».
  - Le cas remonte en tête de la file confirmatrice (cf. §11).
  - Le livreur voit alors « Cas remonté au support ».

  9. Logique des statuts métier

  Quatre statuts partagés par tous les types (Réclamation, Demande, Incident) :

  1. Envoyée — cas créé, non encore pris en charge par la confirmatrice.
  2. En cours de traitement — la confirmatrice l'a pris en main, ou le client a
  répondu à une demande livreur.
  3. Clôturée — traitement terminé, cas résolu.
  4. Refusée — cas non recevable, motif de refus obligatoire.

  Transitions autorisées

  ┌────────────────┬───────────────────────────┬───────────────────────────┐
  │   De → Vers    │      Qui déclenche ?      │         Condition         │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ Envoyée → En   │ Confirmatrice (« Prendre  │                           │
  │ cours de       │ en charge ») ou Client    │ —                         │
  │ traitement     │ (répond à une demande A)  │                           │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ En cours de    │                           │ Action résolutoire        │
  │ traitement →   │ Confirmatrice             │ effectuée (correction     │
  │ Clôturée       │                           │ appliquée, échange créé,  │
  │                │                           │ etc.)                     │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ En cours de    │                           │                           │
  │ traitement →   │ Confirmatrice             │ Motif de refus renseigné  │
  │ Refusée        │                           │                           │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ Envoyée →      │                           │ Cas manifestement         │
  │ Refusée        │ Confirmatrice             │ irrecevable (doublon,     │
  │                │                           │ etc.)                     │
  └────────────────┴───────────────────────────┴───────────────────────────┘

  Pas de réouverture d'un cas clôturé ou refusé — si la situation ressurgit, on
  crée un nouveau cas lié.

  Distinction essentielle : statut métier du cas vs. statut de la commande

  Ce sont deux axes indépendants :
  - Statut du cas (Envoyée / En cours / Clôturée / Refusée) = état du traitement
   administratif.
  - Statut de la commande (Confirmée / En livraison / Livrée / Reportée /
  Retournée) = état logistique.

  La confirmatrice, depuis un cas, peut agir sur les deux séparément.

  11. Logique simplifiée de la file confirmatrice

  Fin du système de score pondéré. Règles de tri strictement ordonnées, faciles
  à expliquer à n'importe quelle confirmatrice :

  1. Urgence — cas dont le motif est Refus client, Autre incident, Colis 
  endommagé, Colis non correspondant. Ils passent en premier.
  2. 3 tentatives ou plus — cas ayant atteint le seuil, quel que soit le motif.
  3. Ancienneté — parmi les cas restants, le plus vieux en premier (FIFO).

  Autrement dit : la file trie en trois passes successives, pas un score
  mystérieux.

  Interface de la file

  - Onglets : À traiter · En cours · Historique · (filtre Type : Réclamation ·
  Demande · Incident).
  - Chaque ligne affiche : badge couleur (urgent / 3 tentatives / normal), type,
   motif, référence commande, client, ancienneté.

  1. Principe général

  SignalR n'est pas une option de confort : c'est le mécanisme principal de 
  cohérence entre les trois acteurs (client, livreur, confirmatrice) et entre
  plusieurs confirmatrices connectées simultanément.

  Trois règles de fond guident toute la conception :

  - Chaque mutation serveur (création de cas, changement de statut, prise en
  charge, correction appliquée, changement de statut commande) déclenche
  obligatoirement un événement SignalR.
  - La source de vérité reste la base de données. SignalR ne transporte jamais
  de données complètes, seulement des identifiants + résumés légers ; le client
  reçoit l'événement puis recharge le détail depuis l'API s'il en a besoin. Ça
  protège contre les races et les messages manqués.
  - La prévention des conflits (deux confirmatrices sur un même cas) est double
  : verrou serveur en base (source de vérité) + notification SignalR (retour
  visuel immédiat).

  6. Mécanisme anti-conflit entre confirmatrices (prise en charge exclusive)

  C'est le point critique du plan temps réel. Deux couches complémentaires :

  Couche 1 — Verrou serveur (source de vérité)

  Le cas possède un champ ConfirmatriceEnCharge (nullable).

  Quand une confirmatrice clique « Prendre en charge », le serveur exécute une
  opération atomique :
  - Condition : le champ doit être null (ou expiré après 30 min).
  - Action : on écrit l'ID de la confirmatrice appelante + horodatage.
  - Si la condition est fausse → erreur 409 « Cas déjà pris en charge par [Nom]
  ».

  Aucun scénario de course à deux gagnants n'est possible : un seul des deux
  appels réussit.

  Couche 2 — Notification SignalR (retour visuel immédiat)

  Dès que le verrou est posé, le serveur émet CasPrisEnCharge vers le groupe
  confirmatrices.

  Toutes les autres confirmatrices :
  - Voient la ligne de file basculer en état grisé avec « Pris en charge par
  [Nom] ».
  - Si elles avaient ce cas ouvert en détail → bannière rouge : « Ce cas est
  maintenant traité par [Nom]. Vous ne pouvez plus intervenir. » Les boutons
  d'action sont désactivés.

  Libération automatique après 30 min

  Un service d'arrière-plan serveur (hosted service) tourne toutes les minutes :
  - Cherche les cas avec ConfirmatriceEnCharge != null + DateDerniereAction < 
  maintenant - 30 min + statut En cours de traitement.
  - Libère : remet ConfirmatriceEnCharge = null, repasse le statut à Envoyée.
  - Émet CasLibere vers le groupe confirmatrices.
  - La confirmatrice qui avait le cas reçoit en plus un événement ciblé pour
  afficher la bannière orange « Cas libéré pour inactivité ».

  Abandon volontaire

  Un bouton « Abandonner le cas » sur le détail permet de libérer volontairement
   (même effet que la libération auto). Utile si la confirmatrice est coupée ou
  réalise qu'un collègue est mieux placé.

  7. Gestion des reconnexions, multi-sessions, pertes réseau

  Trois réalités à couvrir :

  Reconnexion

  Quand la connexion SignalR se coupe et revient (tunnel, 4G qui bascule), les
  événements émis pendant la coupure sont perdus. Règle : à la reconnexion, le 
  client recharge automatiquement l'état depuis l'API.
  - File globale confirmatrice → rechargée en entier.
  - Détail d'un cas ouvert → rechargé.
  - Liste des demandes client → rechargée.

  Ce pattern « événement = signal, API = vérité » rend l'app robuste aux pertes
  réseau.

  Multi-sessions

  Une même confirmatrice peut être connectée sur son mobile et son PC. SignalR
  gère ça nativement (une connexion par session, toutes dans le même groupe
  utilisateur). Les événements arrivent sur les deux écrans, l'état reste
  cohérent.

  Ordre des événements

  Si deux événements arrivent très proches (ex. NouveauCas puis CasPrisEnCharge
  2 secondes après), l'UI doit appliquer les deux dans l'ordre. Solution :
  chaque événement porte un timestamp serveur + le client applique par ordre
  chronologique, et si l'écart entre maintenant et l'événement > 10 secondes, on
   recharge l'API pour être sûr.

  1. Résumé final corrigé

  L'application repose sur deux types métier et rien d'autre :

  - Réclamation = toujours créée par le client vers le support.
  - Demande = toujours créée par le livreur. Selon le motif, la Demande est soit
   visible par le client (action attendue de lui), soit traitée uniquement entre
   le livreur et la confirmatrice.

  Les deux types partagent exactement les 4 statuts métier : Envoyée → En cours 
  de traitement → Clôturée (ou Refusée).

  Un indicateur visuel rouge / vert existe côté client uniquement pour les
  Demandes qui attendent une action de sa part. Cet indicateur n'est pas un
  statut métier.

  Deux mécanismes de routage distincts, à ne jamais confondre :
  - File des cas à traiter (Réclamations et Demandes) = la confirmatrice tire
  elle-même depuis une file globale simple.
  - Distribution des commandes à confirmer = le système pousse chaque nouvelle
  commande vers la confirmatrice la plus disponible via un score clair.

  La reprogrammation est exclusivement côté client. Aucun acteur ne peut choisir
   une nouvelle date à la place du client.

  La communication temps réel (SignalR) est un pilier : chaque mutation serveur
  déclenche un événement qui rafraîchit les écrans concernés sans refresh
  manuel. Détails dans les sections dédiées.

  8. Cas qui vont à la confirmatrice après 3 tentatives

  Téléphone fermé, Client non joignable, Client absent (motifs C).

  Règle de comptage

  Une tentative = un jour d'échec sur une commande. Un seul enregistrement par
  couple (commande, date calendaire) — si le livreur change de statut plusieurs
  fois dans la journée, la tentative n'est comptée qu'une fois.

  Ce que voit le livreur

  - Tentative 1/3 : il coche le motif, enregistre. Rien ne remonte à personne.
  Il voit « Tentative 1 sur 3 enregistrée ».
  - Tentative 2/3 : pareil. Il voit « Tentative 2 sur 3 ».
  - Tentative 3/3 : il coche le motif. Une Demande est automatiquement créée,
  statut Envoyée, badge rouge « 3 tentatives échouées ». Le livreur voit « Cas
  remonté au support — ne plus tenter cette commande aujourd'hui ».

  Ce que voit la confirmatrice

  - La Demande apparaît en tête de file avec le badge rouge « 3 tentatives ».
  - Elle accède au détail et voit l'historique des 3 tentatives (date, heure,
  motif, livreur concerné).
  - Elle contacte le client par téléphone.

  Ce qu'elle peut faire ensuite

  Selon le résultat de son appel :
  - Client atteint, souhaite réessayer → statut cas Clôturée + statut commande
  Confirmée / En livraison (remise en route pour le jour suivant).
  - Client atteint, souhaite reporter → statut cas Clôturée + notification au
  client pour qu'il reprogramme depuis son espace (la reprogrammation est
  déclenchée par lui).
  - Client atteint, refuse la commande → statut cas Clôturée + statut commande
  Retournée.
  - Client reste injoignable → statut cas Clôturée + statut commande Retournée.
  - Dans tous les cas, elle peut ajouter une note interne pour traçabilité.

  10. Logique du changement d'adresse

  Un écran unique et partagé, déclenché dans deux contextes seulement :
  - Création d'une Réclamation avec motif Changement d'adresse.
  - Réponse à une Demande livreur avec motif Adresse incorrecte.

  Jamais pour les autres motifs.

  Contenu de l'écran

  1. En-tête contextuel : rappel de la commande, de l'ancienne adresse, et de
  qui demande la correction.
  2. Champ texte : nouvelle adresse (obligatoire, pour lisibilité humaine).
  3. Bouton principal « Prendre mon adresse actuelle » : déclenche la capture
  GPS, pose un pin automatiquement sur la carte, remplit les coordonnées.
  4. Carte interactive sous le bouton : le client peut déplacer le pin pour
  ajuster.
  5. Affichage textuel des coordonnées sélectionnées (pour confiance).
  6. Bouton de validation « Appliquer » : envoie adresse + latitude + longitude
  au backend.

  Côté confirmatrice

  Elle voit l'ancienne adresse, la nouvelle adresse texte, un mini-aperçu de
  carte centré sur les coordonnées (lecture seule), et un unique bouton
  Appliquer la correction qui propage la valeur au profil et/ou à la commande.

  Technique recommandée

  flutter_map + tuiles OpenStreetMap. Cohérent avec l'utilisation d'OSRM déjà en
   place dans le projet et gratuit.

  ---
  11. Logique de la reprogrammation

  - La reprogrammation est exclusivement client.
  - Aucun bouton, aucun motif, aucune action livreur ou confirmatrice ne permet
  de choisir une nouvelle date à la place du client.
  - Le client la déclenche depuis son espace Commandes (ou depuis le tracking)
  via un motif de Réclamation Demande de reprogrammation.
  - Si un livreur coche Client absent ou Refus client, une notification push
  invite automatiquement le client à reprogrammer lui-même. Le bouton de
  reprogrammation dans son espace devient mis en avant.
  - La confirmatrice peut, dans le cadre du traitement d'un cas, Reporter la
  commande (statut commande Reportée) — mais elle ne fixe pas de date. La
  nouvelle date viendra du client.

  ---
  12. Logique du tracking avec détail complet de commande

  La page Tracking est accessible depuis la liste des commandes côté client.
  Elle centralise tout le contexte nécessaire.

  Contenu

  Bloc 1 — En-tête commande
  - Référence de la commande
  - Statut de la commande (avec pastille couleur)
  - Date de création
  - Mode de livraison / méthode si pertinent

  Bloc 2 — Destinataire
  - Nom complet
  - Téléphone
  - Adresse complète
  - Gouvernorat · Ville

  Bloc 3 — Contenu de la commande
  - Liste des articles (désignation, référence)
  - Quantité par article
  - Total / résumé utile

  Bloc 4 — État de la livraison
  - Timeline des statuts : Confirmée → En livraison → Livrée (ou Reportée /
  Retournée)
  - Horodatage de chaque transition
  - Nom du livreur assigné si disponible

  Bloc 5 — Réclamation liée (affiché si applicable)
  - Motif, date, statut métier
  - Bouton « Voir le détail »

  Bloc 6 — Demande liée (affiché si applicable)
  - Motif, indicateur rouge / vert, bouton « Corriger » si rouge

  Actions contextuelles en bas d'écran
  - « Créer une réclamation » (disponible si des motifs sont éligibles selon le
  statut commande)
  - « Reprogrammer » (uniquement côté client, uniquement si le statut le permet)

  Temps réel

  L'écran s'abonne aux événements SignalR StatutCommandeChange,
  CorrectionAppliquee, NouveauCas, StatutCasChange. Les blocs se rafraîchissent
  sans refresh manuel.

  ---
  13. Interface client : sections et comportement

  4 sections principales

  1. Commandes — liste de ses commandes, pastille statut, accès à Tracking
  2. Réclamations — ses propres cas ouverts et historique
  3. Demandes — cas reçus du livreur (motifs A uniquement)
  4. Profil — ses informations et paramètres

  Détail des sections

  Commandes
  - Liste triée par date décroissante
  - Pastille de statut commande
  - Clic → Tracking (§12)
  - Bouton « Créer une réclamation » si motifs éligibles

  Réclamations
  - Liste triée par date de création décroissante
  - Pour chaque ligne : motif, commande liée, statut métier (pastille), date
  - Pas d'indicateur rouge/vert ici
  - Clic → détail avec historique des actions de la confirmatrice

  Demandes
  - Liste triée par urgence (rouge en premier, puis vert, puis historique gris)
  - Pour chaque ligne : motif, commande liée, indicateur rouge/vert, bouton
  d'action si rouge
  - Bandeau d'alerte en haut : « Vous avez N correction(s) en attente » si au
  moins une rouge
  - Clic sur une ligne rouge → écran de correction (avec carte si adresse)
  - Clic sur une ligne verte → détail d'attente de validation confirmatrice
  - Clic sur historique → détail figé

  Profil
  - Informations personnelles, adresse par défaut, téléphone
  - Déconnexion

  Distinctions à bien afficher

  - Les Réclamations affichent uniquement le statut métier — car c'est le client
   qui les a créées, aucune action n'est attendue de lui après.
  - Les Demandes affichent le statut métier ET l'indicateur rouge/vert. Le
  rouge/vert est ce qui lui parle le plus immédiatement, le statut métier est
  secondaire.

  ---
  14. Interface confirmatrice : sections et comportement

  4 sections principales

  1. Commandes — les commandes à confirmer qui lui sont attribuées + pool
  2. Réclamations — Réclamations clients à traiter
  3. Demandes — Demandes livreurs à traiter
  4. Profil — ses informations, ses statistiques du jour

  Détail des sections

  Commandes
  Cf. section §16 (distribution automatique par score).
  - Onglet « Mes commandes » — les commandes à confirmer qui lui sont attribuées
   par le système.
  - Onglet « Pool global » — les commandes non attribuées ou libérées, qu'elle
  peut prendre manuellement.
  - Onglet « Historique » — ce qu'elle a confirmé récemment.

  Réclamations
  - Onglets : À traiter / En cours / Historique.
  - La file À traiter est la file simplifiée décrite en §15.
  - Chaque ligne : type (Réclamation), motif, commande, ancienneté, badges
  (urgent, 3 tentatives si applicable).
  - Clic → détail complet.

  Demandes
  - Mêmes 3 onglets.
  - Chaque ligne : type (Demande), motif, commande, ancienneté, badges,
  indicateur d'état client pour les Demandes A (en rouge si le client n'a pas
  encore corrigé, en vert si il vient de corriger).
  - Clic → détail complet.

  Profil
  - Nom, rôle, statut (En ligne / Indisponible)
  - Statistiques du jour : commandes confirmées, cas traités, temps moyen
  - Bouton « Se mettre en pause » (permet de ne plus recevoir de nouvelles
  attributions)

  Écran détail d'un cas

  Tout sur une seule page, 5 blocs + 2 barres d'actions.

  Blocs
  1. Client — nom, téléphone, email, adresse, GPS si disponible, historique (nb
  commandes, nb réclamations/demandes), bouton Appeler le client
  2. Livreur (seulement si Demande) — nom, téléphone, position GPS, bouton
  Appeler le livreur
  3. Commande — référence, articles, montants, mode paiement, mode livraison,
  timeline statuts
  4. Cas — type (Réclamation ou Demande), motif, description si Autre incident,
  photos si applicable, correction proposée (pour Demandes A si client a
  répondu), historique des tentatives si applicable
  5. Note interne — zone de texte libre visible uniquement par les
  confirmatrices

  Barre 1 — Actions sur le cas
  - Prendre en charge (si Envoyée) · Abandonner (si elle le détient)
  - Appliquer la correction (Demandes A uniquement)
  - Créer un échange (Réclamations Colis endommagé / Colis non correspondant)
  - Clôturer · Refuser (avec motif)
  - Ajouter / modifier note interne

  Barre 2 — Actions sur la commande
  - Remettre en livraison (statut Confirmée ou En livraison)
  - Reporter (statut Reportée)
  - Retourner (statut Retournée)

  Les deux barres sont séparées visuellement pour rappeler que ce sont deux axes
   distincts. La confirmatrice peut cliquer sur l'un sans toucher à l'autre.

  Flexibilité mise en avant

  La confirmatrice peut systématiquement :
  - Clore un cas sans modifier la commande (ex. l'information est traitée mais
  la livraison continue).
  - Modifier la commande sans clore le cas (ex. elle remet en livraison mais
  laisse le cas ouvert pour surveillance).
  - Modifier les deux indépendamment dans n'importe quel ordre.

  ---
  15. File confirmatrice simplifiée (Réclamations + Demandes)

  Il s'agit de la file de traitement des cas. À ne pas confondre avec la 
  distribution des commandes à confirmer (§16).

  Principe

  Une file globale unique partagée par toutes les confirmatrices, quel que soit
  le gouvernorat du client. Pas de score pondéré, pas de complexité.

  Tri en trois passes successives

  1. Urgents d'abord — cas dont le motif est Refus client, Autre incident, Colis
   endommagé, Colis non correspondant.
  2. 3 tentatives ou plus ensuite — cas ayant atteint le seuil.
  3. Anciens ensuite (FIFO) — le plus vieux en premier.

  Facile à expliquer à n'importe quelle confirmatrice. Facile à débugger.

  Interaction

  - Chaque confirmatrice voit la file triée.
  - Bouton Prendre en charge sur chaque ligne → verrou serveur + statut En cours
   de traitement.
  - Bouton global Suivant en haut → système ouvre le cas de plus haute priorité
  non verrouillé.
  - Si une autre confirmatrice détient déjà le cas → bannière rouge « Pris en
  charge par [Nom] » et actions désactivées.
  - Libération automatique si inactivité > 30 min → le cas revient dans la file.

  Temps réel

  La file se met à jour automatiquement via les événements SignalR NouveauCas,
  CasPrisEnCharge, CasLibere, SeuilTentativesAtteint.

  ---
  16. Distribution des commandes à confirmer entre confirmatrices

  Mécanisme distinct et indépendant de la file de cas. Il s'agit ici des
  nouvelles commandes créées qui attendent d'être confirmées (statut En 
  attente).

  Principe général

  - Agence globale : une confirmatrice peut confirmer une commande de n'importe
  quel gouvernorat.
  - Attribution automatique : chaque nouvelle commande est poussée vers la
  confirmatrice la plus disponible via un score simple.
  - Après confirmation : la commande part chez les livreurs du gouvernorat du 
  client (pas de changement de ce côté).

  Deux volets qui cohabitent

  1. Mes commandes à confirmer (push) — commandes auto-attribuées à la
  confirmatrice. Elle les voit dans son onglet dédié dès leur création.
  2. Pool global (pull) — commandes non attribuées ou libérées. Toute
  confirmatrice peut en prendre manuellement. Permet la reprise si l'attribuée
  est absente ou lente.

  Timer de libération

  Si une commande attribuée n'est pas traitée en 30 minutes, elle est libérée :
  - Elle retourne dans le Pool global.
  - Elle est automatiquement réattribuée à la prochaine confirmatrice la plus
  disponible.
  - La confirmatrice concernée reçoit une notification « Commande libérée pour
  inactivité ».

  Admin

  L'admin voit une vue de supervision : nombre de commandes attribuées à
  chacune, taux de libération, temps moyen de traitement. Permet de rééquilibrer
   manuellement si nécessaire.

  ---
  17. Score simple de disponibilité des confirmatrices

  Formule claire, tenant en 4 lignes. Calculée en temps réel à chaque nouvelle
  commande.

  Score = BaseConnexion
        − (ChargeEnCours × 15)
        − (Fatigue × 1)
        + (AttenteDepuisDernièreAttribution × 2)

  Détail des facteurs

  Facteur: BaseConnexion
  Définition: 100 si la confirmatrice est en ligne ET active (dernière action <
    10 min), sinon 0 (exclue du tirage)
  Plafond: 100
  ────────────────────────────────────────
  Facteur: ChargeEnCours
  Définition: Nombre de commandes attribuées non encore confirmées
  Plafond: —
  ────────────────────────────────────────
  Facteur: Fatigue
  Définition: max(0 ; nbConfirméesAujourdhui − 25)
  Plafond: Pénalise au-delà de 25 confirmations dans la journée
  ────────────────────────────────────────
  Facteur: AttenteDepuisDernièreAttribution
  Définition: Minutes depuis la dernière commande qu'elle a reçue
  Plafond: Plafonné à 30 minutes

  Exemple concret

  ┌──────────────┬──────────┬───────┬────────────┬────────┬───────┬──────┐
  │ Confirmatric │ Connecté │ Charg │ Confirmées │ Attent │ Calcu │ Scor │
  │      e       │    e     │   e   │  aujourd'h │   e    │   l   │  e   │
  │              │          │       │     ui     │        │       │      │
  ├──────────────┼──────────┼───────┼────────────┼────────┼───────┼──────┤
  │              │          │       │            │        │ 100 − │      │
  │ Amira        │ Oui      │ 2     │ 10         │ 5 min  │  30 − │ 80   │
  │              │          │       │            │        │  0 +  │      │
  │              │          │       │            │        │ 10    │      │
  ├──────────────┼──────────┼───────┼────────────┼────────┼───────┼──────┤
  │              │          │       │            │        │ 100 − │      │
  │ Sonia        │ Oui      │ 0     │ 40         │ 15 min │  0 −  │ 115  │
  │              │          │       │            │        │ 15 +  │      │
  │              │          │       │            │        │ 30    │      │
  ├──────────────┼──────────┼───────┼────────────┼────────┼───────┼──────┤
  │ Lina         │ Non      │ 0     │ 0          │ —      │ 0     │ 0    │
  └──────────────┴──────────┴───────┴────────────┴────────┴───────┴──────┘

  → Sonia remporte la prochaine commande : libre et en attente depuis longtemps.

  Logique d'attribution

  1. À la création d'une commande, calculer le score de chaque confirmatrice
  active.
  2. Sélectionner celle avec le score le plus élevé. En cas d'égalité : tirage
  aléatoire.
  3. Attribuer la commande (champ ConfirmatriceAttribuée) + émission d'un
  événement SignalR « CommandeAttribuée » vers cette confirmatrice.
  4. Si aucune confirmatrice disponible (toutes hors ligne) → la commande reste
  dans le Pool global, visible dès qu'une confirmatrice se connecte.

  Pourquoi cette formule ?

  - Exclusion stricte des hors-ligne (score à 0).
  - Punition forte de la charge (coefficient 15) — l'objectif est d'étaler le
  travail.
  - Punition légère de la fatigue (coefficient 1) — on ne bloque pas quelqu'un,
  on oriente légèrement les suivantes ailleurs.
  - Protection anti-famine (coefficient 2) — une personne peu sollicitée remonte
   naturellement.

  Quatre facteurs suffisent. Pas besoin d'aller plus loin en V1.

  V1 simple, évolutions possibles plus tard

  - V1 : 4 facteurs, auto-attribution.
  - V2 éventuelle : ajouter vitesse moyenne de traitement, ou ajustement manuel
  par l'admin.
  - V3 éventuelle : affinage par heure de la journée, jour de semaine.

  On ne part qu'avec V1.
