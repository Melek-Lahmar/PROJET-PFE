  client_orders_screen.dart only provides CustomerOrdersProvider +
  ClientClaimsProvider when pushing the tracking route →
  ProviderNotFoundException (the user perceives this as the "create reclamation"
   button being broken).

  Bug 2 (Backend — 500 on list/create): MapListAsync builds an EF query with
  pieces.Contains(l.DO_Piece.Trim().ToUpper()) against nullable string columns.
  EF Core 8 sometimes fails to translate Trim().ToUpper() against nullable
  nvarchar inside Contains, throwing at execution time → 500. Also the
  controller wraps only specific exceptions; everything else bubbles up as a raw
   500 with no message body, so the Flutter UI shows nothing useful.

  Bug 3 (EF model): F_RECLAMATION_MESSAGE has both ReclamationId (intended FK)
  and a shadow F_RECLAMATIONId (auto-created by EF because the Messages
  navigation isn't explicitly mapped). The shadow column is harmless
  functionally but pollutes the schema; explicitly configuring the relationship
  eliminates the issue cleanly.

  - Les 4 statuts sont ambigus. « Approuvée » et « Traitée » se confondent. Je
  propose plus clair :
    - Envoyée → En cours → Résolue ou Refusée.
  Si tu tiens à tes 4 mots, définis-les explicitement : Approuvée = décision
  prise, Traitée = action terminée. Sinon l'utilisateur ne comprend pas la
  nuance.
  - "Autre motif" ne doit pas redevenir un champ texte ouvert. Garde une
  description courte (≤ 300 caractères), obligatoire, plus photo optionnelle.
  - Photo "importante mais modifiable" : trop flou. Règle binaire par motif :
  obligatoire (colis endommagé, colis non correspondant) ou optionnelle partout
  ailleurs. La modification tant que non traitée reste, mais règle de
  "importance" disparaît.
  - Changement d'adresse : map + description ne suffit pas en soi. La
  confirmatrice doit valider avant que la nouvelle adresse écrase l'ancienne. Le
   client propose, la confirmatrice tranche.
  - Ne pas créer deux tables parallèles (demandes client, signalements livreur).
   C'est la même entité avec un champ Source = CLIENT / LIVREUR. Sinon tu
  dupliques la logique.

  4. Ce qu'il faut supprimer

  - Le chat complet : le hub SignalR des réclamations, la table des messages,
  les deux écrans de chat Flutter (client + confirmatrice), les deux providers
  de chat, les endpoints /messages.
  - Les messages système et internes (SYSTEM, IsInternal) : sans chat, ils ne
  servent plus.
  - Le motif libre sur le formulaire de création : remplacé par une liste
  fermée.
  - Tout le code de broadcast de messages en temps réel (tu n'en as plus besoin
  pour un changement de statut, une mise à jour simple suffit).

  5. Ce qu'il faut ajouter

  - Motifs client (liste fermée) : changement adresse, changement numéro,
  reprog, annulation, non reçu, endommagé, non correspondant, comportement
  livreur, autre.
  - Motifs livreur (liste fermée) : injoignable, téléphone éteint, numéro
  incorrect, absent, adresse incorrecte/incomplète/introuvable, demande report,
  refuse la commande, colis endommagé, tentative échouée, autre.
  - Champ Source (CLIENT ou LIVREUR) sur la demande.
  - Photos : 1 à 5 images max, stockées comme actuellement, liées à la demande.
  - Correction proposée : quand le motif est "changement adresse" ou "changement
   numéro", le client saisit directement la nouvelle valeur dans la demande. Si
  approuvée, la confirmatrice l'applique d'un clic.
  - Table des avis : note 1-5, commentaire court, une entrée par commande.
  - État de relance de l'avis : date de dernière invite, nombre de rappels, pour
   savoir quand réafficher la popup.
  - Alerte dans le tracking client : quand un signalement livreur attend une
  correction du client (ex : "Le livreur n'a pas trouvé ton adresse — corrige-la
   maintenant").
  - Actions rapides confirmatrice sur le détail d'une demande : Approuver,
  Traiter, Refuser, Appeler le client, Appliquer la correction, Ouvrir la map.

  6. Logique finale proposée

  Client : sur le tracking, bouton « J'ai un problème » → choix d'un motif →
  photo ou correction si demandé → envoi. Onglet « Mes demandes » : liste avec
  statut, clic = détail.

  Livreur : sur le détail d'une livraison en cours, bouton « Signaler un
  problème » → choix d'un motif → photo/note optionnelle → envoi. Il voit ses
  signalements dans son historique.

  Confirmatrice : une seule page « Demandes & signalements ». Filtres : statut,
  date, commande, gouvernorat, source, motif. Clic = détail complet : commande,
  client, motif, photos, correction proposée. Boutons d'action directs. Chaque
  changement de statut est visible immédiatement côté client sur son tracking.

  Admin : même vue que la confirmatrice, mais sur tous les gouvernorats, plus
  quelques stats.

  Popup d'avis : quand une commande passe en LIVRE, on la marque "avis attendu".
   À la prochaine ouverture de l'app (ou au retour sur l'écran commandes), si au
   moins 24h se sont écoulées depuis le dernier rappel, on affiche la popup.
  Avis envoyé → terminé pour cette commande. Fermée → on retentera plus tard
  (max 3 rappels). Elle revient pour chaque nouvelle commande livrée.

  9. Risques ou points d'attention

  - Perte de l'historique des chats existants : archive la table avant de la
  dropper, ne la supprime pas d'un coup.
  - Ambiguïté des statuts : rédige un libellé humain à côté de chaque statut
  pour que l'utilisateur comprenne (« Envoyée = on a reçu ta demande », « En
  cours = on s'en occupe », etc.).
  - Upload photo qui échoue sur mauvais réseau : ne bloque pas l'envoi de la
  demande. Enregistre la demande, réessaie la photo en arrière-plan, ou permet
  de la rajouter après.
  - Correction d'adresse appliquée trop vite : impose la validation
  confirmatrice avant écriture. Un clic "Appliquer" explicite, pas une
  approbation implicite.
  - Popup d'avis abusive : max 1 par session d'app, 24h mini entre deux rappels,
   3 rappels max par commande, puis on lâche.
  - Plusieurs demandes/signalements sur la même commande : groupe-les par
  commande dans la liste confirmatrice, sinon c'est le bazar.
  - Droits d'accès : re-valide que le client voit uniquement ses demandes, le
  livreur uniquement ses commandes, la confirmatrice uniquement son gouvernorat.
   À tester après la refonte.
  - Renommage éventuel F_RECLAMATION → F_DEMANDE : utile pour clarifier, mais
  c'est cosmétique — ne bloque pas la refonte pour ça, fais-le en tout début si
  tu y tiens.

  cette commande ; au 3e cumulé, on crée la demande et on lie les 3 tentatives.

  Règle "une seule demande active par commande × motif"

  Tant qu'une demande est ouverte (Envoyée ou En cours) pour un motif donné sur
  une commande donnée, les nouvelles tentatives du même motif n'en créent pas 
  une deuxième. Elles incrémentent le compteur et s'ajoutent à l'historique de
  la demande existante.

  Si la demande devient Résolue ou Refusée, un nouveau problème du même type sur
   la même commande peut recréer une demande (nouveau cycle).

  Auto-résolution quand la livraison réussit

  Si le livreur finit par livrer (statut → LIVRE) alors qu'une demande était
  ouverte pour cette commande, la demande passe automatiquement en Résolue avec
  la note "Livraison effectuée par le livreur". Pas de cas zombie, pas de
  travail inutile pour la confirmatrice.

  Affichage côté client (important)

  Même avant la création d'une demande, le client doit voir les tentatives dans
  son tracking, sinon il reste dans le noir.

  - 1re tentative "téléphone éteint" → bandeau sur le tracking : "Le livreur a
  tenté de te joindre — rallume ton téléphone. Prochaine tentative prévue."
  - 2e tentative → "Deuxième tentative. Rallume ton téléphone."
  - 3e tentative → "Troisième tentative. Support contacté." + demande visible
  dans "Mes demandes".

  Ça conserve l'esprit de ton idée (ne pas harceler la confirmatrice), tout en
  gardant le client informé pour qu'il puisse corriger de son côté avant que ça
  escalade.

  Affichage côté confirmatrice

  Quand la demande arrive au 3e essai, le détail montre clairement les 3 
  tentatives horodatées (date, heure, position GPS si dispo, motif, livreur).
  Elle voit toute l'histoire d'un coup.

  Seuil configurable

  Le seuil "3" est un paramètre du backend, pas une constante hardcodée. On peut
   plus tard l'ajuster par motif (2 pour "client absent", 3 pour "téléphone
  éteint"), mais pour la v1 un seuil unique de 3 suffit.

  Ce qui est solide — à garder sans toucher

  - Confirmatrice centrale avec actions groupées (appeler, appliquer correction,
   changer statut commande, résoudre, refuser).

  Ça, c'est propre. Rien à changer.

  Les 8 points flous à clarifier maintenant

  1. Comptage des tentatives : par motif ou par commande ?

  Problème : si le livreur fait 1× "téléphone éteint", 1× "client absent", 1×
  "injoignable", est-ce 1 tentative par motif (donc aucune demande créée) ou 3
  tentatives cumulées sur la commande (donc escalade au 3e) ?

  Ma recommandation : compter globalement par commande, pas par motif. 3 échecs
  quel que soit le motif "différé" → escalade. Raison : 3 échecs sur la même
  commande = vrai problème, peu importe l'étiquette précise.

  2. Cooldown entre tentatives du livreur

  Problème : un livreur pressé peut faire 3× "téléphone éteint" en 2 minutes et
  déclencher une demande inutile.

  Ma recommandation : imposer un délai minimum de 20 minutes entre 2 tentatives
  sur la même commande. Si le livreur essaie trop vite, son écran refuse et
  affiche "Attends avant de réessayer". Simple à coder, évite les faux
  déclenchements.

  3. Photo obligatoire côté livreur pour "colis endommagé"

  Problème : si le livreur signale un colis endommagé sans preuve visuelle, la
  confirmatrice n'a rien pour trancher.

  Ma recommandation : photo obligatoire côté livreur pour les motifs colis 
  endommagé uniquement. L'écran de changement de statut impose l'ajout avant
  validation. Même règle binaire que pour le client.

  4. Remise en livraison après correction

  Problème : la confirmatrice corrige l'adresse. Faut-il que la commande repasse
   automatiquement en CONFIRME (pour être re-planifiée) ou qu'elle le fasse à la
   main ?

  Ma recommandation : explicite, pas automatique. Dans le détail demande, bouton
   "Remettre en livraison" qui remet la commande en CONFIRME. Raison : elle peut
   vouloir vérifier, appeler le client, coordonner avec le livreur avant de
  relancer. Pas d'automatisme invisible.

  5. Le livreur voit-il les corrections confirmatrice ?

  Problème : si la confirmatrice change l'adresse, le livreur doit savoir sinon
  il repasse à la mauvaise adresse.

  Ma recommandation : oui, le livreur voit la mise à jour sur la commande, sans
  détail de la demande. Un petit bandeau sur son écran livraison : "Adresse mise

  1. Comptage des tentatives — version corrigée (ta règle)

  Règle finale : 1 tentative = 1 journée d'échec, pas 1 changement de statut.

  - Dans la même journée, le livreur peut appeler 2-3 fois, faire plusieurs
  tentatives d'approche, basculer le statut plusieurs fois. Tout ça compte pour
  1 seule tentative journalière.

  motif "soft" (injoignable, téléphone éteint, absent…), on enregistre 1 
  tentative pour ce jour.
  - Le lendemain, s'il remet un statut différé → 2e tentative.
  - Le jour 3 → 3e tentative → escalade automatique à la confirmatrice.

  "tentative" par couple (commande, date du jour) pour les motifs différés. Si
  le livreur revient changer le statut le même jour, on met à jour l'entrée
  existante (dernière heure, dernier motif), on n'en crée pas une nouvelle.

  règle "1 par jour" absorbe naturellement le cas du livreur impatient. On
